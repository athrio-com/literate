# The three levels

LF is a three-level algebra. Everything in the framework fits one
of these levels:

```
Concept      (interface — declares what something IS)
    ↓ realised by
Trope        (class — describes HOW the Concept is done in LF;
                     prose-first, Effect Schema-backed)
    ↓ instantiated by consumers as
Authored     (file in consumer's corpus, matching the Trope's shape)
```

## Concept

A Concept is LF's metalanguage entry for a kind of thing. It
declares schema, invariants, and relationships. Concepts are pure
declaration — they ship as TypeScript modules at
`@literate/concept-<id>` and export a `Concept<I>` value with an
Effect Schema for the instance shape.

[Browse all concepts →](/docs/concepts)

## Trope

A Trope is the canonical realisation of a Concept. Each Trope ships
as `@literate/trope-<id>` and exports a `Trope<C>` value carrying:

- A real module reference to the Concept it realises.
- Real module references to other Tropes it depends on (cross-Trope
  references are TS imports; missing references are compile errors).
- Optional subkinds (typed refinements, e.g. ADR within decisions).
- Optional members (typed primitives, e.g. category members).
- Optional Effect `Layer` for runtime services.
- A prose-first MDX body describing the realisation.

[Browse all tropes →](/docs/tropes)

## Authored instance

A consumer's actual file — an ADR they wrote, a session log, a
custom Term — is an Authored instance of some Trope. Authored
instances live in the consumer's `corpus/`. They are not shipped
by LF and not themselves leaves of the algebra.

## Composition

A consumer's `package.json` declares the installed Trope set under
the `"literate"` key. The CLI imports each Trope package, walks the
typed dependency graph, and materialises `.literate/` — a vendored
snapshot containing the Concept and Trope prose plus a
`manifest.json` describing the resolved graph.
