### Goal 1 — Add `proseSchema` to `Trope<C>` and validate during weave

**Status:** Completed
**Category:** code-from-prose
**Topic:** Extend `Trope<C>` in `@literate/core`'s `kinds.ts` with
`proseSchema: Schema.Schema<ParsedMdx, any>` where `ParsedMdx` is
the mdast-typed shape produced by remark parsing. Parse each
vendored Trope's `prose.mdx` with remark; validate structural
shape against the Trope's declared `proseSchema` during weave.
Failures become typed `ProseSchemaViolation { tropeId, path,
expected, got }` surfaced in aggregate at weave end (matching the
session-end validator pattern — continue for other Tropes, collect
all failures, raise together).

`proseSchema` is the **ship-side analogue** of `Concept<I>`'s
`instanceSchema`: Concepts constrain what consumers author as
instances; Tropes constrain what LF (or consumers via vendored
edits) author *as Tropes*. Same machinery, two scopes (§6 of the
exploration protocol).

**Upstream:** ADR-025 (mechanical weave); ADR-026 §4 (CLI–Trope
binding is bundled-from-source, so the CLI has access to the
canonical Trope objects including their `proseSchema`);
exploration §6 + §13.3.

**Acceptance:**
- `proseSchema: Schema.Schema<ParsedMdx, any>` required on
  `Trope<C>` and its `TropeDefinition<C>` in `packages/core/src/kinds.ts`.
  `ParsedMdx` exported from `@literate/core`.
- Every seeded Trope in `registry/tropes/` carries a `proseSchema`.
- The weave step parses each vendored Trope's prose (sigil
  stripped), validates against `proseSchema`, and surfaces typed
  `ProseSchemaViolation` violations together as
  `ProseSchemaViolations` if any — file writing is blocked on
  violations; consumer fixes and re-runs.
- Smoke test covers one structurally-correct Trope and one
  structurally-broken Trope (missing a required H2).

**Notes:**
- `proseSchema` validates *structural markers that make a Trope a
  Trope*, not every paragraph. Starting minimal: require top-level
  H1 + the set of H2 section slugs the authored prose currently
  carries.
- The ADR-024 §3 generated-file sigil applies to weave output
  (`.literate/LITERATE.md`), not authored seeds — `registry/tropes/
  <id>/prose.mdx` is never sigilled. The weaver still strips any
  HTML-comment sigil prefix before parsing, as a defensive
  invariant.

## Decisions Made

No ADRs authored this session (per Person directive: architecture
is settled at 27 ADRs across 5 days; P3 is implementation). The
following design choices land inline:

- **`proseSchema` is required on every `Trope<C>`**, not optional.
  Every Trope must declare what shape its authored prose takes;
  the minimal escape hatch is a permissive schema (`requireMdxStructure({ h1: '…', h2Slugs: [] })`).
- **`ParsedMdx` is a structural type in `@literate/core`**, not an
  import of `@types/mdast`. Core stays lean (no remark deps); the
  CLI owns the actual parse via `unified + remark-parse + remark-mdx`.
  Seed authors compose `Schema.Schema<ParsedMdx>` via the core-
  provided `requireMdxStructure` helper; custom refinements compose
  via `Schema.filter`.
- **Validation aggregates before writing.** Weave collects every
  `ProseSchemaViolation` across all vendored Tropes and raises a
  single `ProseSchemaViolations` error *before* `.literate/LITERATE.md`
  is written — mirrors `trope-session-end`'s `SessionEndIncomplete`
  pattern. Consumer fixes the flagged seeds and re-runs.
- **Sigil-strip is defensive**, not load-bearing. ADR-024 §3's
  generated-file sigil applies to weave *output* only; authored
  seeds never carry it. The weaver still strips any
  `<!-- GENERATED by LF … -->` HTML-comment prefix before parsing,
  so a consumer who accidentally pastes a sigil into a vendored
  `prose.mdx` doesn't get a schema failure attributable to the
  sigil rather than their edit.
- **The bundled Trope registry lives in the weaver**, not in
  `trope-bindings.ts`. The weaver's `TROPE_REGISTRY` (`session-start`
  → `sessionStartTrope`, `session-end` → `sessionEndTrope`) is the
  single id → Trope lookup; `trope-bindings.ts` keeps its narrow
  role as the build-time re-export point (ADR-026 §4).
- **Goal 2 (Variant `proseSchema` symmetry) deferred.** Zero
  Variant-using Tropes at v0.1 means zero authoring sites; re-open
  the symmetry question at the first such Trope rather than
  pre-emptively add API surface.

### Discovered in this session

- **`Schema.declare` + `Schema.filter` is the idiomatic
  composition** for authoring prose schemas. `Schema.filter`
  returns string-or-true; string is the error message that
  surfaces through `Schema.decodeUnknownEither` into our
  `ProseSchemaViolation.got`. No custom issue types needed.
- **The consumer-extension Trope path is unvalidated at v0.1.**
  `TROPE_REGISTRY` is a closed map keyed on bundled Trope ids;
  consumer-authored Tropes under `.literate/extensions/tropes/<id>/`
  have no bundled `proseSchema` lookup. This is the correct
  boundary — LF can't ship schemas for Tropes it doesn't author.
  When consumer Tropes carry their own `index.ts` with a
  `proseSchema`, a future enhancement could dynamically source
  that field (ADR-026 §4 currently rejects dynamic import; the
  question re-opens if/when that policy shifts).
- **Sigil-strip is cheap insurance.** The authored seeds never
  carry the sigil, but writing the regex-based strip before
  parse cost ~5 lines and means a consumer who copy-pastes a
  sigilled `LITERATE.md` into a vendored `prose.mdx` gets a
  schema error about their actual edit rather than a
  sigil-artefact false positive.
- **Bundle size grew from the remark pull-in** (~800 kB → 1671 kB).
  Acceptable at v0.1 since `@literate/cli` is the sole npm
  artefact and users run it via `bunx`/`npx`; bundle size is
  not a hot path. A future release-engineering session may
  evaluate `micromark` directly (remark's lower-level core) or
  a lighter markdown parser if the bundle footprint becomes a
  concern.
- **`ParsedMdx` as a structural supertype of mdast's `Root`**
  kept core free of `@types/mdast` + remark type deps. The
  weaver parses with real remark (producing a real mdast Root)
  and decodes against our loose schema; the structural match
  is exact at runtime. If a future Schema needs richer mdast
  type information (e.g., inline code nodes), importing mdast
  types at that point is a non-breaking addition.
