# Metadata — typed head-of-file metadata

A **Metadata block** is the typed key→value record that authored
prose files carry at the head of the file, above any narrative
body. The block declares operational facts about the file:
identity (a content hash), the typed Domain it belongs to, its
Layer, its current operational status, optional cross-references.

The block is a **substrate primitive**: every authored prose
surface that wants typed head-of-file metadata uses this same
shape. LFMs are the principal consumer today; future typed prose
surfaces (specs, chapters, registry seed prose) can attach
metadata blocks under the same Concept.

## Wire form

The canonical wire form is the leaf directive `::metadata{...}`
declared by `:lfm[annotation-substrate]{hash=9a6b8081}`:

```markdown
::metadata{id=9a6b8081, disposition={base:'Protocol', scope:'annotation-substrate'}, layer={kind:'protocol', path:'protocol', holds:'domains'}, domain=annotation-substrate, status=Reconciled}

# Annotation substrate

LF authored prose carries typed cross-references …
```

A leaf directive's body is empty by definition; all information
lives in the `{key=val, key=val, ...}` attributes. The directive
is one line; long values quote with the standard CommonMark
attribute syntax (single quotes around the value).

The previous wire form is the YAML `---` fenced block:

```markdown
---
id: 9a6b8081
disposition: { base: 'Protocol', scope: 'annotation-substrate' }
layer: { kind: 'protocol', path: 'protocol', holds: 'domains' }
domain: annotation-substrate
status: Reconciled
---

# Annotation substrate

LF authored prose carries typed cross-references …
```

Both forms parse to the same typed `Metadata` value. Reconcile
migrates the legacy form to the directive form mechanically;
once the corpus is fully migrated, YAML parsing retires.

## Shape

```typescript
type Metadata = Record<string, string>
```

The substrate keeps values opaque. Each `key=val` pair becomes
one `(key, value)` entry; no parsing of the value happens at the
Metadata layer. A consumer (e.g. the `lfm` Concept) reads its
own typed values out of the record.

## Typical fields (LFM consumer)

When the consumer is an LFM, the metadata block typically
carries:

- `id` — short content hash (8 hex chars).
- `disposition` — the typed `Disposition` JSON-ish literal.
- `layer` — the typed `Layer` JSON-ish literal.
- `domain` — the LFM's Domain name within its Layer.
- `status` — operational `LFMStatus` (`Reconciled` /
  `Drifted` / `Pending` / `Unverified`).
- `dependencies` — optional JSON array of soft-link LFM ids.

These keys are not enforced by the Metadata Concept itself; they
are enforced downstream by the `lfm` Concept's `LFMSchema`. The
Metadata substrate's job is to provide the typed key→string
record; the LFM's job is to interpret the values against its
own typed contract.

## Why retire YAML?

Three motivations:

1. **One annotation surface.** The annotation substrate
   declares `:`-prefixed directives as the unified surface for
   typed substrate content. Keeping YAML for frontmatter splits
   the surface in two; folding frontmatter into `::metadata`
   collapses everything onto the directive grammar.
2. **No second parser.** Today the substrate carries a
   minimal YAML parser (line-by-line key/value extraction).
   Folding into the directive form lets the `remark-directive`
   parser already wired into the weaver pipeline carry the
   load.
3. **Better composition.** The directive form composes with
   inline / leaf / container directives in the same prose;
   YAML frontmatter is a special-cased preamble that does not.

## Consumers

- **LFMs** at `corpus/manifests/<layer>/<domain>.md` — every
  LFM declares its metadata via `::metadata{...}` (or, until
  fully migrated, the legacy `---` form). The `lfm` Concept's
  `LFMSchema` interprets the values.
- **Future typed prose surfaces** — specs, chapters, registry
  seed prose can attach metadata blocks under the same
  Concept.

## Resolution

`literate learn metadata` resolves to this Concept seed and
returns its prose body plus the bound `metadata` Trope at
`registry/tropes/metadata/`.
