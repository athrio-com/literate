# ADR

An **ADR** (Architecture Decision Record) is the
*frozen* and *one-shot* prose artefact recording an
architectural commitment. ADRs are append-only post-Accept
(IMP-6); the `Status:` line is the sole mutable part. Use
ADRs for tooling commitments ("we use library X"),
append-only amendments to prior ADRs, or decisions that
benefit from an unambiguous historical anchor.

This Concept declares the typed shape of an ADR as a passive
type surface at v0.1. ADR authoring is currently a
markdown-on-disk activity guided by the `## Decisions` block
of `corpus/CLAUDE.md` (IMP-2 ADR path). A future `adr-flow`
Trope can use this Schema for typed draft-and-gate flow.

## Shape

```typescript
interface ADR {
  readonly _tag: 'ADR'
  readonly number: number
  readonly title: string
  readonly status: ADRStatus              // from concepts/adr-status
  readonly tags: ReadonlyArray<Tag>       // from concepts/tag
  readonly date?: string
  readonly supersedes?: ReadonlyArray<number>
  readonly supersededBy?: ReadonlyArray<number>
}
```

`ADRStatus` and `Tag` are imported from sibling Concepts.
Composition is the point: types reference types.

## On-disk shape

An ADR file at `corpus/decisions/ADR-NNN-slug.md` carries the
following header convention:

```markdown
# ADR-NNN — Title

**Status:** Accepted
**Date:** YYYY-MM-DD
**Tags:** `#process` `#protocol`
**Supersedes:** ADR-MMM (optional)

## Context
...
## Decision
...
## Consequences
...
```

## Mutability

Per IMP-6, the body of an Accepted ADR is **frozen**. Only
the `Status:` line is mutable, and only via restricted
transitions (see `adr-status`). Supersession is recorded by
setting the prior ADR's `Status:` to
`Superseded by ADR-NNN[ — note]`; the prior ADR's body is not
edited.

## Tag requirements

Per `corpus/CLAUDE.md`, every ADR must carry at least one
tag drawn from the consumer's authored tag set. The Tag
Concept ships with a brand-typed shape; the closed-set
membership is enforced by each consumer separately (see
`tag/concept.mdx`).
