# Implication

An **Implication** is a *soft Goal* — something an exploration
or any work surfaced that has authorial weight but hasn't been
adjudicated into a Goal yet. Parallel machinery to Goal,
softer state, different gating profile.

## Shape

```typescript
interface Implication {
  readonly _tag: 'Implication'
  readonly id: string
  readonly status: 'Surfaced' | 'Promoted' | 'Filed' | 'Dismissed'
  readonly rationale?: string  // required when status = 'Dismissed'
}
```

The Schema enforces the **rationale-required-on-Dismissed**
invariant via a refinement on the struct, not via downstream
validation.

```typescript
const ImplicationStruct = Schema.Struct({
  _tag: Schema.Literal('Implication'),
  id: Schema.String,
  status: Schema.Union(
    Schema.Literal('Surfaced'),
    Schema.Literal('Promoted'),
    Schema.Literal('Filed'),
    Schema.Literal('Dismissed'),
  ),
  rationale: Schema.optional(Schema.String),
})

const ImplicationSchema = ImplicationStruct.pipe(
  Schema.filter((value) =>
    value.status !== 'Dismissed' || (value.rationale ?? '').trim() !== '',
  {
    message: () =>
      'Implication: rationale required when status is Dismissed',
  }),
)
```

## The four statuses

- **`Surfaced`** — the Implication has been raised but not yet
  dispositioned. Default state when an Implication is added to
  a session. Surfacing is ungated — anyone (Person or agent)
  can record an Implication mid-session by appending an entry
  to the session's `## Implications` section.
- **`Promoted`** — the Implication has been promoted to a Goal.
  Promotion is gated (the Goal it became carries its own
  Status / Category and follows the standard Goal lifecycle).
  The Implication entry retains the link to the promoted Goal.
- **`Filed`** — the Implication has been filed for later
  consideration. Filing produces a memo under
  `corpus/memos/<slug>.md` (typed handoff for a future
  session); the Implication entry retains the memo path.
  Filing is gated.
- **`Dismissed`** — the Implication has been explicitly retired
  without action. Dismissal requires a rationale string
  (Schema-enforced); the rationale is the journaled reason
  the Implication should not consume future attention.
  Dismissal is gated.

`Surfaced` is the *non-terminal* state. The other three are
terminal: at session-end every Implication must carry a
terminal status (`Promoted | Filed | Dismissed`). The
`session-end` Trope's validator refuses to close a session
with a non-terminal Implication.

## Implication vs Goal

Goal and Implication are parallel machinery for adjudicating
attention. They share the lifecycle shape (gated decisions
across a closed status set) but differ in:

- **Strength of intent.** A Goal is committed work for the
  current session. An Implication is something the work has
  surfaced that *might* become a Goal — here, in a future
  session, or not at all.
- **Status set.** Goals: `Active | Completed | Superseded by
  Goal N | Abandoned`. Implications: `Surfaced | Promoted |
  Filed | Dismissed`.
- **Where they live.** Goals live in the session's `## Goals`
  block; Implications live in the session's `## Implications`
  block (introduced by this Concept; backward-compatible
  absent for older logs).
- **Promotion path.** An Implication promoted becomes a Goal
  (this session, if the Person Accepts) or seeds a new
  session's planned Goal set (if Filed for later).

The Concept is the typed surface for what previous LF prose
called "loose ends," "follow-ups," and "things we noticed."
Naming and typing them lifts them from journal text to
addressable entities the gate can act on.

## When Implications surface

Any Mode can surface an Implication, but `Exploring` is the
common case: a deliberation surfaces three possible directions
and the Person picks one as the session's Goal; the other two
record as `Surfaced` Implications. `Weaving` can surface them
too — drafting an ADR may reveal that a related ADR needs
amendment; the related-ADR amendment becomes a Surfaced
Implication that the Person triages at session-end.

`Tangling` Mode tends not to surface Implications (the work is
deterministic derivation from accepted prose); when it does,
they typically come from drift discovered between prose and
code — those become Filed Implications targeting a future
prose-update session.

## Schema-level invariant

The `rationale-required-on-Dismissed` invariant is enforced
at the Schema level. An attempt to construct
`{ status: 'Dismissed', rationale: undefined }` fails Schema
validation; the typed error surfaces as a
`ParseError` from `Schema.decodeUnknown` or as a typed
constructor failure from `Schema.encode`. Downstream code
does not need to re-check.

## Session-end validation

The `session-end` Trope's `validateStep` extends to:

```
if any Implication.status === 'Surfaced':
  return Effect.fail(SessionEndIncomplete {
    sessionPath, missing: ['Implication[<id>].terminal-status']
  })
```

Failures surface in the session-end error channel as the
existing `SessionEndIncomplete` aggregate (with a
`Implication[<id>].terminal-status` line in `missing`). Future
work may introduce a separate `ImplicationNotTerminal`
TaggedError per the spec; v0.1 piggybacks on the existing
aggregate to stay backward-compatible with the session-end
prose-schema.
