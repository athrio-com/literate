# Subkinds

A subkind refines a Trope by narrowing its instance Schema. Subkinds
are *not* alternative realisations of the Concept — they are
variations within a single Trope.

The v0.1 example: the **ADR** subkind of the `decisions` Trope. It
fixes `subkind: 'ADR'` in the schema and provides ADR-specific
prose (`# ADR-NNN — title` header, `Date / Status / Tags` lines,
`Context / Decision / Consequences` sections).

A future book project might add an `SDR` subkind (story decisions);
a product team might add `PDR` (product decisions). Each subkind is
exported from the parent Trope's package as a `Subkind<C>` value.

## Shape

```ts
import { Schema } from 'effect'
import type { Subkind } from '@literate/core'
import { DecisionsConcept } from '@literate/concept-decisions'

export const ADRInstanceSchema = Schema.Struct({
  subkind: Schema.Literal('ADR'),
  number: Schema.Number,
  // ...
})

export const ADRSubkind: Subkind<typeof DecisionsConcept> = {
  _tag: 'Subkind',
  id: 'ADR',
  version: '0.1.0',
  realises: DecisionsConcept,
  instanceSchema: ADRInstanceSchema as never,
  prose: proseFrom(import.meta.url, './ADR.mdx'),
}
```

The parent Trope lists its subkinds:

```ts
export const decisionsTrope: Trope<typeof DecisionsConcept> = {
  // ...
  subkinds: [ADRSubkind],
}
```

## Composition

When the CLI compiles a consumer's `.literate/`, each Trope's
subkinds are materialised at
`.literate/tropes/<trope>/subkinds/<subkind>.mdx`. The shipped
`manifest.json` records the subkind ids alongside the Trope.
